### [2026-03-26] 历史日志：设备写权限检测

**当时目标：**
实现设备写权限检测 API，要求既能准确反映设备可写状态，又不能对设备状态产生副作用，尤其要适应 macOS 下 U 盘默认挂载导致整盘节点不可写的实际情况。

**最初设想：**
一开始以为只要用 os.access 或尝试 open 设备节点就能判断可写性，预期只要设备挂载且权限允许就能写，没考虑到 macOS 下 U 盘默认挂载后整盘节点其实是 busy 状态，根本无法直接写入。

**现实冲突与权衡：**
实际测试发现，macOS 下 U 盘插入后会自动挂载，/dev/diskN 处于 busy 状态，os.open 直接报错。只有卸载后，/dev/diskN 才能写入镜像。但如果检测阶段自动卸载，用户体验很差，且有副作用（比如 Finder 里盘突然消失）。

权衡后决定：
- 检测阶段绝不自动卸载，保持无副作用。
- 只检测当前挂载状态和权限，发现 busy 或只读时，明确提示“需先卸载再写入”。
- API 返回 (bool, dict)，dict 里详细说明不可写的原因，便于 UI/CLI 友好提示。

**实现细节与感受：**
实现过程中，发现单纯用 os.access 很不靠谱，很多情况下权限没问题但设备 busy。最终采用 os.open 捕获 OSError，结合 diskutil info 判断挂载状态，才算比较靠谱。
写 API 时，反复思考“检测不做副作用操作”原则，避免了很多潜在风险。实际测试时，发现很多用户（包括自己）会误以为“检测不可写=坏盘”，所以返回结构里必须详细区分“busy/只读/权限不足”等不同原因。

**对当前新架构的映射：**
- 设备当前是否挂载、是否 busy、是否只读，应由 `core/platform/mac/device_probe.py` 提供原始探测结果
- “目标设备当前能否进入执行阶段” 应由 `core/planner/validation.py` 或执行前预检决定
- “真正开始执行前是否自动卸载” 应由 `core/executor` 调用 `core/platform/mac/disk_adapter.py` 处理
- API / CLI 只负责把这些状态和原因呈现给用户，不直接承载底层判断

**实际感受：**
macOS 下磁盘管理的“自动挂载”机制和 Linux/Windows 差异很大，很多直觉在这里都不成立。做安全工具时，流程设计必须极度克制，不能为了一步到位而牺牲用户可控性。每一步都要考虑“如果我是用户，看到盘突然消失会不会慌”，所以检测和操作分得很细。API 结构也要为 UI/CLI 留足弹性，不能一刀切。

**原有逻辑流程设想 vs. 调整后流程决策**

【原有设想流程】
1. 列出所有可用设备
2. 检查所有设备的写权限，自动过滤不可写设备
3. 用户只需选择“可写”设备，直接进入写盘
4. 检测和写入流程合并，追求一步到位

【历史上调整后的流程】
1. 列出所有可用设备供用户选择（不预判可写性）
2. 等待用户明确选择目标设备
3. 用户选择后，检测该设备的写权限和状态：
	- 如果设备未 busy 且可写，直接进入写盘流程
	- 如果设备 busy（如 macOS 下已挂载），则自动尝试卸载，再检测一次可写性
	- 卸载失败或依然不可写，返回详细原因，提示用户手动处理
4. 检测和写入流程解耦，但“检测可写性”本身允许自动卸载（因否则永远不可写），其余操作仍需用户确认
5. 整个流程每一步都可追溯，CLI/GUI 可根据检测和卸载结果做细致提示

【变化与原因】
- 原设想自动过滤不可写设备，用户只需点一次，体验简单但不透明，且有副作用风险。
- 现流程强调“先选设备再检测”，检测阶段如遇 busy 会自动卸载（否则永远不可写），其余操作仍需用户确认，最大限度避免误操作。
- 这样虽然多一步，但用户对每个环节都心里有数，遇到 busy/只读等情况能获得明确解释和操作建议。
- 自动卸载只在“检测可写性”时发生，且有详细提示，避免用户困惑。
- 也便于后续扩展，比如 CLI/GUI 可以根据检测和卸载结果做更细致的交互和提示，流程更灵活。

## 当前补充

在新架构里，这份日志更适合作为以下设计提醒：
- 不要让 `platform/mac` 的探测逻辑偷偷做破坏性动作
- 不要让 `planner` 和 `executor` 混淆“状态判断”和“执行动作”
- 不要让 CLI 直接决定平台层策略
